# 기능 명세서 작성 가이드

> 이 문서는 `/docs` 디렉토리에 기능 명세서를 작성할 때의 규칙과 템플릿을 정의한다.
>
> **워크플로우**: 명세서 작성 → 구현 → 구현 완료 후 명세서 삭제 → 반복

---

## 1. 워크플로우

```
① 기능 기획/설계 시 → docs/에 명세서 작성
② 구현 진행 중     → 명세서를 참조하며 개발
③ 구현 완료        → 명세서 삭제 (git에서 제거)
④ 새 기능 시작     → ①부터 반복
```

### 파일 규칙

| 항목 | 규칙 |
|------|------|
| **위치** | `/docs/` 디렉토리 루트 |
| **파일명** | 케밥케이스, 기능을 설명하는 이름 (예: `openclaw-cli-backend-setup.md`) |
| **언어** | 한국어 (코드/타입/경로는 영어) |
| **삭제 시점** | 해당 기능의 모든 Phase가 구현 완료되고 정상 동작 확인 후 |

### 이 가이드만 영구 보존

`SPEC-GUIDE.md`는 삭제하지 않는다. 나머지 명세서는 구현 완료 후 삭제한다.

---

## 2. 명세서 구조 (필수 섹션)

모든 명세서는 아래 구조를 따른다. 각 섹션의 목적과 작성 기준을 상세히 설명한다.

```markdown
# {기능명} 명세서

> **목적**: {한 문장으로 이 기능이 무엇을 하는지}
>
> **작성일**: YYYY-MM-DD

---

## 1. 배경과 동기
## 2. 현재 상태 분석
## 3. 설계
## 4. 상세 구현
## 5. 테스트
## 6. 구현 단계
## 7. 주의사항
## 8. 기대 효과
## 9. 파일 변경 요약
```

## 3. 각 섹션 작성 기준

### 3.1 제목 + 헤더 블록

```markdown
# {기능을 명확히 설명하는 제목} 명세서

> **목적**: {이 기능이 하는 일을 한 문장으로. "~한다" 형식으로 끝낸다}
>
> **작성일**: YYYY-MM-DD
```

**규칙**:
- 제목은 기능의 정체를 바로 알 수 있게 작성
- 목적은 반드시 한 문장. 기술 용어 사용 가능하지만 행위의 주체와 결과가 명확해야 한다
- 작성일은 최초 작성일 기준

**좋은 예**:
> **목적**: AI Hub에서 OpenClaw을 Claude CLI 백엔드로 구동하도록 설정하는 기능을 구현한다.

**나쁜 예**:
> **목적**: OpenClaw 설정 기능 추가

### 3.2 섹션 1: 배경과 동기

**이 섹션이 답해야 하는 질문**: "왜 이걸 만들어야 하는가?"

**필수 포함 항목**:

#### 1.1 현재 문제
- 지금 시스템이 어떻게 동작하는지 **구체적으로** 설명
- ASCII 다이어그램이나 코드 블록으로 현재 흐름을 시각화
- 사용자 관점에서 어떤 문제가 발생하는지 명시

```markdown
### 1.1 현재 문제

현재 미션 시스템은 **모든 작업을 Claude CLI로만 실행**한다.

\```
사용자 → "내일 아침 10시에 뉴스 분석해줘"
       → Claude CLI 실행
       → "저는 스케줄을 설정할 수 없습니다" 반환
\```
```

#### 1.2 목표 상태
- 이 기능이 완성된 후의 동작을 **동일한 시나리오**로 보여줌
- 현재 문제와 대비되도록 작성

```markdown
### 1.2 목표 상태

\```
사용자 → "내일 아침 10시에 뉴스 분석해줘"
       → 오케스트레이터가 분석
       → OpenClaw cron add → 완료 보고
\```
```

#### 1.3 선택의 이유 (해당되는 경우)
- 여러 방법 중 이 방법을 선택한 이유
- 비교표나 트레이드오프 분석 포함

### 3.3 섹션 2: 현재 상태 분석

### 3.5 섹션 4: 상세 구현

**이 섹션이 답해야 하는 질문**: "코드를 어떻게 작성할 것인가?"

**이 섹션이 명세서의 핵심이다.** 구현자(사람 또는 AI)가 이 섹션만 보고 코드를 작성할 수 있어야 한다.

**필수 포함 항목 (파일별로)**:

- **파일 경로**: 정확한 절대 경로가 아닌 프로젝트 루트 기준 상대 경로
- **전체 코드 또는 핵심 코드 블록**: 주요 함수/컴포넌트의 **실제 구현 코드**
- **코드 내 주석**: 핵심 로직에 한국어 주석
- **변경 파일의 경우**: 변경 전/후 diff를 명확히

```markdown
### 4.1 `openclaw-config.ts` — 설정 관리 모듈

\```typescript
// packages/electron-app/src/lib/openclaw-config.ts

export interface OpenClawConfig {
  gateway: {
    mode: 'local';
    port: number;
    // ...
  };
}

/** 현재 설정 파일 읽기. 없으면 null */
export function readConfig(): OpenClawConfig | null {
  // 구현...
}
\```
```

**코드 블록 작성 규칙**:

| 규칙 | 설명 |
|------|------|
| 파일 경로 주석 | 코드 블록 첫 줄에 `// 파일경로` 주석 필수 |
| 언어 태그 | 코드 블록에 `typescript`, `tsx`, `json` 등 언어 명시 |
| export 명시 | 외부에서 사용할 함수/타입은 `export` 포함 |
| 에러 처리 | try/catch 패턴 포함 (실제 구현과 동일하게) |
| 타입 완전성 | `any` 사용 금지, 실제 타입 정의 포함 |

**UI 컴포넌트의 경우 추가**:
- 레이아웃 설명 (어떤 요소가 어디에 위치하는지)
- 상태 관리 (useState, useCallback 등)
- 사용자 인터랙션 흐름 (클릭 → API 호출 → 결과 표시)
- 스타일은 기존 패턴(`S` 객체)에 맞춰 작성

### 3.6 섹션 5: 테스트

**이 섹션이 답해야 하는 질문**: "구현한 코드가 정상 동작하는지 어떻게 검증하는가?"

**모든 명세서는 반드시 테스트 코드를 포함한다.** 테스트 없이 구현 완료로 간주하지 않는다.

**필수 포함 항목**:

#### 테스트 파일 목록
- 신규/수정할 테스트 파일 경로
- 각 테스트 파일이 검증하는 대상 모듈

```markdown
| 테스트 파일 | 대상 모듈 | 설명 |
|------------|----------|------|
| `src/__tests__/openclaw-config.test.ts` | `openclaw-config.ts` | 설정 읽기/쓰기, CLI 확인 |
| `src/__tests__/openclaw-config-api.test.ts` | `api/openclaw/config/route.ts` | 설정 API 엔드포인트 |
```

#### 테스트 케이스 명세
- 각 함수/API별로 **정상 케이스**와 **실패 케이스**를 모두 포함
- `describe` / `it` 블록 구조로 작성
- mock이 필요한 경우 mock 대상과 방법을 명시

```markdown
\```typescript
// src/__tests__/openclaw-config.test.ts

describe('readConfig', () => {
  it('openclaw.json이 없으면 null을 반환한다', () => { /* ... */ });
  it('유효한 설정 파일이면 파싱된 객체를 반환한다', () => { /* ... */ });
  it('잘못된 JSON이면 null을 반환한다', () => { /* ... */ });
});

describe('checkClaudeCli', () => {
  it('claude가 PATH에 있으면 available: true를 반환한다', () => { /* ... */ });
  it('claude가 없으면 available: false를 반환한다', () => { /* ... */ });
});
\```
```

#### 테스트 작성 규칙

| 규칙 | 설명 |
|------|------|
| **파일 위치** | `src/__tests__/` 디렉토리에 `{기능명}.test.ts` |
| **네이밍** | `describe`는 함수/컴포넌트명, `it`은 "~하면 ~한다" 형식 (한국어) |
| **모킹** | 외부 의존성(파일 I/O, CLI 실행, fetch)은 mock 처리. mock 대상을 명세서에 명시 |
| **경계값** | 빈 값, null, 잘못된 타입, 인증 실패 등 엣지 케이스 포함 |
| **API 테스트** | 인증 필요 여부, 요청/응답 스키마, HTTP 상태 코드 검증 |
| **UI 테스트** | 필요한 경우만. 상태 변화와 사용자 인터랙션 흐름 중심 |

**테스트 범위 기준**:
- 신규 모듈: 내보내는 모든 함수에 대해 최소 정상/실패 각 1개 이상
- 신규 API 라우트: 각 HTTP 메서드에 대해 정상/인증실패/유효성실패 케이스
- 수정 파일: 변경된 로직에 대한 테스트 추가 (기존 테스트가 있으면 케이스 추가)

### 3.7 섹션 6: 구현 단계

**이 섹션이 답해야 하는 질문**: "어떤 순서로 구현하는가?"

**필수 포함 항목**:

- **Phase 분류**: 의존 관계에 따라 Phase 분리
- **각 Phase 내 순서**: 파일 생성/수정 순서
- **Phase별 테이블**: 순서, 작업, 파일 컬럼

```markdown
### Phase 1: 인프라 (파일 3개)

| 순서 | 작업 | 파일 |
|------|------|------|
| 1-1 | 설정 관리 모듈 생성 | `src/lib/openclaw-config.ts` |
| 1-2 | 설정 API 생성 | `src/app/api/openclaw/config/route.ts` |
| 1-3 | 게이트웨이 매니저 수정 | `src/lib/gateway-manager.ts` |

### Phase 2: UI (파일 1개)

| 순서 | 작업 | 파일 |
|------|------|------|
| 2-1 | 설정 탭 추가 | `src/app/settings/page.tsx` |
```

**규칙**:
- Phase 간에는 의존 관계가 있어야 함 (Phase 2는 Phase 1 완료 후에만 가능)
- Phase 내의 작업은 가능한 독립적으로 (병렬 구현 가능하게)
- 각 Phase 완료 시 **동작 가능한 상태**여야 함 (부분 배포 가능)
- **테스트 작성은 반드시 별도 Phase로 포함**한다 (보통 마지막 또는 각 Phase 직후)
- **프로덕트 전반에 큰 변화가 있는 기능은 README 업데이트 Phase를 포함**한다 (아래 4.5 참고)

### 3.8 섹션 7: 주의사항

**이 섹션이 답해야 하는 질문**: "구현 시 무엇을 조심해야 하는가?"

**카테고리별로 정리**:

| 카테고리 | 포함 내용 |
|----------|----------|
| **보안** | API 키 노출, 인증, 권한 관련 |
| **호환성** | 기존 기능과의 충돌, 하위호환 |
| **경로/환경** | 파일 경로, 환경변수, OS 차이 |
| **타이밍** | 비동기 처리, 재시작 대기, 레이스 컨디션 |
| **엣지 케이스** | 설정 파일 이미 존재, 네트워크 끊김, 프로세스 종료 등 |
| **한계/향후 과제** | 현재 범위에서 다루지 않는 것, 향후 개선 방향 |

```markdown
### 6.1 보안
- `openclaw.json`에 API 키를 넣지 않는다
- loopback에서만 `auth.mode: "none"` 허용

### 6.2 기존 설정 파일 존재 시
- 이미 openclaw.json이 있는 경우 모델만 변경
- Phase 1에서는 전체 덮어쓰기로 단순화
```

## 4. 작성 원칙

### 4.1 구체성 원칙

```
❌ "적절한 에러 처리를 추가한다"
✅ "readConfig()가 null을 반환하면 HTTP 400으로 '설정 파일이 없습니다' 에러를 반환한다"

❌ "UI를 추가한다"
✅ "설정 페이지의 탭 네비게이션에 'OpenClaw' 항목을 추가하고,
    모델 선택 라디오 버튼 3개 + '설정 적용' 버튼으로 구성한다"

❌ "API를 만든다"
✅ "GET /api/openclaw/config → { ok, status, supportedModels }
    POST /api/openclaw/config { model } → { ok, config, gateway }"
```

### 4.2 코드 우선 원칙

설명으로 애매한 것은 **코드로 보여준다**.

```
❌ "설정 객체에 게이트웨이 모드와 에이전트 모델 정보를 포함한다"

✅
\```typescript
interface OpenClawConfig {
  gateway: { mode: 'local'; port: number; bind: 'loopback' };
  agents: { defaults: { model: string } };
}
\```
```

### 4.3 현재 코드 기반 원칙

- 기존 코드를 **반드시 읽고** 현재 패턴에 맞춰 작성
- 스타일 객체가 `S`로 정의되어 있으면 그 패턴을 따름
- API 라우트가 `getSession(req)` 패턴이면 동일하게
- DB 접근이 `Settings.get()`이면 동일하게

### 4.4 자기완결성 원칙

명세서 하나만 읽으면 구현할 수 있어야 한다.

- 다른 문서를 참조할 때는 경로를 명시 (예: "`docs/orchestrator-design.md` 참조")
- 외부 라이브러리 사용 시 패키지명과 import 경로 포함
- 환경변수 사용 시 이름과 기본값 명시

### 4.5 README 동기화 원칙

프로덕트 전반에 영향을 주는 큰 변화를 구현할 때는 **반드시 프로젝트 README를 업데이트**한다.

**README 업데이트가 필요한 경우**:
- 새로운 시스템 구성 요소가 추가될 때 (예: 새 백엔드, 새 서비스)
- 아키텍처 흐름이 달라질 때 (예: 인증 방식 변경, 데이터 흐름 변경)
- 설치/실행 방법이 바뀔 때 (예: 새 환경변수, 새 의존성, 새 빌드 단계)
- 사용자가 알아야 할 주요 기능이 추가될 때

**README 업데이트가 불필요한 경우**:
- 내부 리팩토링 (외부 동작 변경 없음)
- 버그 수정
- UI 미세 조정
- 기존 기능의 작은 개선

**명세서에 포함하는 방법**:
- 구현 단계에 "README 업데이트" Phase를 추가
- 파일 변경 요약에 README 파일을 포함
- 변경할 섹션과 추가할 내용을 구체적으로 명시

```markdown
### Phase N: 문서화

| 순서 | 작업 | 파일 |
|------|------|------|
| N-1 | 아키텍처 다이어그램에 새 구성 요소 추가 | `README.md` |
| N-2 | 환경변수 섹션에 신규 변수 추가 | `README.md` |
```
